# Deployment

## Infrastructura

CuibOS rulează pe un cluster **K3s** (Kubernetes lightweight) pe **Hetzner Cloud**, cu edge routing prin **Cloudflare**.

```
┌─────────────────────┐     ┌──────────────────────┐
│  Cloudflare CDN     │     │  Vercel              │

│  (api.cuibos.ro)    │     │  (app.cuibos.ro)     │
└──────────┬──────────┘     └──────────────────────┘
           │
           │ CF Access Tunnel
           ▼
┌──────────────────────────────────────┐
│  Hetzner Cloud                       │
│  ┌────────────────────────────────┐  │
│  │  K3s Cluster                   │  │
```

## CI/CD

Fiecare microserviciu are un pipeline **GitHub Actions**:

1. **Push pe `main`** → trigger workflow
2. **Build** → Docker image
3. **Push** → GitHub Container Registry (ghcr.io)
4. **Deploy** → `kubectl apply` pe clusterul K3s

### Gateway

Gateway-ul Cloudflare Worker are workflow separat:
1. Push pe `main` → `wrangler deploy`

### Frontend

Frontend-ul (app.cuibos.ro) se deployează automat pe **Vercel** la push pe `main`.

## JWKS Key Rotation

`auth-service` este sursa de adevăr pentru cheile JWT.

Fluxul curent:

1. Se rotește perechea RSA în secretul Kubernetes `auth-jwt-signing`
2. `auth-service` expune JWKS-ul curent la `/auth/v1/jwks.json`
3. Workflow-ul de deploy pentru `auth-service` publică JWKS-ul live în Cloudflare KV
4. Gateway-ul citește din KV în regim normal
5. Dacă apare un `kid` necunoscut, gateway-ul face refresh din `auth-service` și resincronizează KV

Pentru rotație fără întrerupere, JWKS-ul publicat trebuie să conțină cheia nouă și cheia anterioară până când expiră toate token-urile semnate cu cheia veche.
