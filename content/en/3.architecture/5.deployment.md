## External Services

| **Stripe** | Payments and subscriptions | stripe.com |
| **Cloudflare Workers** | Edge gateway | cloudflare.com |

| **Cloudflare Access** | Secure tunnel | cloudflare.com |
| **Vercel** | Frontend hosting | vercel.com |
| **Resend** | Email delivery | resend.com |

## CI/CD

Each microservice has a **GitHub Actions** pipeline:

1. **Push to `main`** вҶ’ trigger workflow
2. **Build** вҶ’ Docker image
3. **Push** вҶ’ GitHub Container Registry (ghcr.io)
4. **Deploy** вҶ’ `kubectl apply` on K3s cluster

### Gateway

The Cloudflare Worker gateway has a separate workflow:
1. Push to `main` вҶ’ `wrangler deploy`

### Frontend

The frontend (app.cuibos.ro) deploys automatically on **Vercel** on push to `main`.

## JWKS Key Rotation

`auth-service` is the source of truth for JWT signing keys.

Current flow:

1. Rotate the RSA key pair in the `auth-jwt-signing` Kubernetes secret
2. `auth-service` exposes the live JWKS at `/auth/v1/jwks.json`
3. The `auth-service` deploy workflow publishes that live JWKS into Cloudflare KV
4. The gateway reads from KV in steady state
5. If an unknown `kid` appears, the gateway refreshes from `auth-service` and re-syncs KV

For zero-downtime rotation, the published JWKS must include both the new key and the previous key until all tokens signed with the old key have expired.
