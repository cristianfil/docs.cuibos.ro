# Events

CuibOS uses an **event-driven** architecture with **transactional outbox** for inter-service communication.

## Pattern: Transactional Outbox

```
┌──────────────────────────────────┐
│         DB Transaction           │
│                                  │
│  1. INSERT/UPDATE business data  │
│  2. INSERT event into outbox     │
│                                  │
│         COMMIT                   │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│     Outbox Poller (async)        │
│                                  │
│  3. Read unpublished events      │
│  4. Publish to RabbitMQ          │
│  5. Mark as published            │
└──────────────────────────────────┘
```

Benefits:
- **Atomicity** — data and event are saved in the same transaction
- **Reliability** — if RabbitMQ is unavailable, events remain in the outbox
- **Idempotency** — consumers can process the same event multiple times

## The event_outbox table

Each service that produces events has an `event_outbox` table:

| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `routingKey` | string | Routing key (e.g., `properties.property.created`) |
| `payload` | JSON | Event body |
| `tenantId` | UUID | Source tenant |
| `correlationId` | string | Original request ID |
| `publishedAt` | timestamp | Null until published |
| `createdAt` | timestamp | Creation time |

## RabbitMQ Exchange

All events are published to the `events` exchange (topic exchange). Consumers subscribe using routing keys with wildcards.

## Event Categories

### Property Events

| Routing key | Payload | Produced by |
|---|---|---|
| `properties.property.created` | `{tenantId, propertyId, name, propertyType, locked}` | property-service |
| `properties.property.updated` | `{tenantId, propertyId, name, propertyType, locked}` | property-service |
| `properties.property.deleted` | `{tenantId, propertyId, deletedAt}` | property-service |
| `properties.property_user.activated` | `{tenantId, propertyId, userId, role}` | property-service |
| `properties.property_user.revoked` | `{tenantId, propertyId, userId}` | property-service |

**Consumers:** billing-service (usage tracking)

### Vehicle Events

| Routing key | Payload | Produced by |
|---|---|---|
| `vehicles.vehicle.created` | `{tenantId, vehicleId}` | vehicles-service |
| `vehicles.vehicle.updated` | `{tenantId, vehicleId}` | vehicles-service |
| `vehicles.vehicle.deleted` | `{tenantId, vehicleId}` | vehicles-service |

**Consumers:** billing-service (usage tracking)

### Auth Events

| Routing key | Payload | Produced by |
|---|---|---|
| `auth.tenant.plan_changed` | `{tenantId, plan, previousPlan}` | auth-service |
| `auth.user.registered` | `{tenantId, userId, email}` | auth-service |

**Consumers:** billing-service, property-service

## Inter-service Flows

### Property creation → billing usage

```
property-service                    billing-service
     │                                   │
     │  properties.property.created      │
     ├──────────────────────────────────▶│
     │                                   │
     │                    Update PropertyUsageState
     │                    Recompute subscription usage
     │                                   │
```

### Paid registration flow

```
Stripe → billing-service (webhook)
         │
         │  Create SignupCheckout record
         │
         ▼
app → auth-service (register-paid)
      │
      │  Claim SignupCheckout
      │  Associate Stripe customer + subscription with tenant
      │
      ▼
auth-service → billing-service (event: tenant.plan_changed)
```
